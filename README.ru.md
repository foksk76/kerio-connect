# Kerio Connect Lab

Воспроизводимое Docker-окружение для лабораторного запуска Kerio Connect на Debian 13.

[![Docker Lab CI](https://github.com/foksk76/kerio-connect/actions/workflows/docker-image.yml/badge.svg)](https://github.com/foksk76/kerio-connect/actions/workflows/docker-image.yml)

> **Статус проекта:** лабораторный проект для безопасной проверки поведения Kerio Connect перед боевым использованием.

> **Языковая политика:** `README.md` — основной README на английском языке. `README.ru.md` — основной русский перевод для лабораторной работы и быстрого знакомства с проектом. Сохраняйте переключатель языка первой строкой в обоих файлах.

## Зачем нужен этот репозиторий

Kerio Connect обычно устанавливается как обычный Linux-пакет или виртуальный appliance. Это подходит для ручного администрирования, но неудобно, когда инженеру нужно повторяемое локальное окружение для проверки первичной настройки, поведения после перезапуска, лицензирования и экспорта логов.

Этот репозиторий собирает локальную Docker-лабораторию вокруг официального Debian-пакета Kerio Connect. Вендорские бинарные файлы остаются внутри образа, а состояние, лицензии, почтовое хранилище и логи сохраняются в Docker volumes.

## Семейство проектов

Этот репозиторий является частью семейства проектов **Kerio Connect Monitoring & Logging**:

1. [kerio-connect](https://github.com/foksk76/kerio-connect) — воспроизводимое лабораторное окружение Kerio Connect
2. [kerio-logstash-project](https://github.com/foksk76/kerio-logstash-project) — пайплайн парсинга и хранения Kerio syslog
3. [kerio-syslog-anonymizer](https://github.com/foksk76/kerio-syslog-anonymizer) — детерминированная анонимизация реальных логов для безопасной публикации

## Место репозитория в общей схеме

Этот репозиторий предоставляет воспроизводимую лабораторную исходную систему Kerio Connect.

```text
kerio-connect -> Kerio syslog -> kerio-logstash-project -> Elasticsearch / Kibana
                              -> примеры логов -> kerio-syslog-anonymizer
```

Связанные репозитории дополняют друг друга:

- `kerio-connect` поднимает воспроизводимое лабораторное окружение Kerio Connect.
- `kerio-logstash-project` парсит, нормализует, проверяет и сохраняет Kerio syslog в ELK-ориентированном пайплайне.
- `kerio-syslog-anonymizer` готовит реальные логи к безопасной публикации, сохраняя детерминированные значения для корреляции.

## Основной сценарий использования

1. Инженер подготавливает `.env` и выбирает, как будет предоставлен официальный Debian-инсталлятор Kerio Connect.
2. `docker compose build` собирает Debian 13 образ с Kerio Connect и скриптами-обвязками репозитория.
3. `docker compose up -d` запускает лабораторный Kerio Connect и монтирует постоянные Docker volumes.
4. Инженер открывает административный интерфейс на порту `4040` и проходит первичный мастер настройки вендора.
5. Kerio Connect работает как исходная система для проверок почтового сервиса и опционального экспорта Syslog.
6. Результат проверяется через состояние контейнера, открытые порты, статус сервиса внутри контейнера и admin endpoint.

## Архитектура / Роли компонентов

1. **Исходная система**: Kerio Connect внутри лабораторного контейнера.
2. **Обвязка сборки**: `Dockerfile` устанавливает официальный Debian-пакет Kerio Connect из `artifacts/`, `KERIO_DOWNLOAD_URL` или публичного архива Kerio.
3. **Оркестрация запуска**: `docker-compose.yml` публикует admin и mail-порты, задает CPU, память, healthcheck и volumes.
4. **Слой постоянного хранения**: Docker volumes сохраняют конфигурацию, почтовое хранилище, лицензионные файлы и логи.
5. **Скрипты / инструменты**: `scripts/` подготавливает состояние, по возможности настраивает путь логов, запускает Kerio Connect и выполняет healthcheck.
6. **Экспорт логов**: Kerio Connect можно настроить на отправку выбранных логов во внешний Syslog-приемник, например в связанный Logstash-проект.

## Требования

### Программное обеспечение

- ОС: Linux-хост с поддержкой Docker; Debian 13 — текущий целевой и локально проверенный путь.
- Docker: обязателен.
- Docker Compose plugin: обязателен.
- Сеть: нужна при `KERIO_AUTO_DOWNLOAD=1`, если в `artifacts/` нет локального инсталлятора.
- Опциональный локальный артефакт: один официальный Kerio Connect Debian `.deb` в `artifacts/` для offline или pinned-сборки.

### Аппаратные ресурсы

- CPU: минимум вендора для `1-20` пользователей — `2 core`, `1 GHz`, `64-bit`; практическая рекомендация для лаборатории — `4 vCPU`.
- RAM: минимум вендора — `4 GB`; практическая рекомендация — `8 GB`.
- Диск: минимум вендора — `40 GB` свободного места для хранилища сообщений и backup; практическая рекомендация — `60 GB`.

### Проверенные версии

| Компонент | Версия | Примечания |
|---|---|---|
| Debian | 13 (Trixie) | текущая цель репозитория |
| Docker Engine | 26.1.5+dfsg1 | проверено локально |
| Docker Compose | 2.26.1-4 | проверено локально |
| Kerio Connect | 10.0.9 | проверено в этой лаборатории; точная сборка может измениться, если включен auto-download |

## Быстрый старт

> Короткий путь: собрать и запустить локальную лабораторию Kerio Connect, проверить состояние контейнера, открытые порты и административный интерфейс.

План работ:

- скопировать `.env.example` в `.env` и проверить параметры под ваш хост;
- собрать образ лабораторного Kerio Connect;
- запустить сервис `kerio-connect`;
- проверить healthcheck контейнера и опубликованные порты;
- открыть начальный admin endpoint.

### 1. Клонируйте репозиторий

```bash
git clone https://github.com/foksk76/kerio-connect.git
cd kerio-connect
ls README.md docker-compose.yml .env.example
```

Если все хорошо:

- текущая директория является корнем репозитория;
- присутствуют `README.md`, `docker-compose.yml` и `.env.example`.

### 2. Подготовьте окружение

```bash
cp .env.example .env
scripts/enable-git-hooks.sh
docker compose config >/tmp/kerio-connect.compose.txt
sed -n '1,40p' /tmp/kerio-connect.compose.txt
```

**Что можно изменить**

- `KERIO_ADMIN_PORT`: измените, если порт `4040` уже занят на хосте.
- `KERIO_SMTP_PORT`: оставьте `25` только если порт свободен; иначе используйте значение вроде `2525`.
- `KERIO_VERSION_LABEL`: задайте, если хотите закрепить версию из архива Kerio.
- `KERIO_DOWNLOAD_URL`: задавайте только если нужен явный официальный URL Linux DEB.
- `KERIO_DOWNLOAD_SHA256`: задайте, если нужна проверка checksum загружаемого DEB.
- `KERIO_MEMORY_LIMIT` и `KERIO_CPUS`: измените, если на лабораторном хосте другие лимиты ресурсов.

**Что важно**

- Оставьте `KERIO_AUTO_DOWNLOAD=1`, если хотите, чтобы сборка сама нашла текущий публичный Kerio Connect Linux DEB в официальном архиве.
- Поместите ровно один официальный Kerio Connect Debian installer в `artifacts/`, если нужна offline или pinned-сборка.
- Остановите локальный MTA или переназначьте `KERIO_SMTP_PORT`, если хост уже слушает порт `25`.

```bash
systemctl stop postfix || true
ss -ltn '( sport = :25 )'
```

Если все хорошо:

- `docker compose config` завершается без ошибок;
- сгенерированная конфигурация содержит сервис `kerio-connect`;
- порт `25` на хосте свободен или намеренно переназначен в `.env`.

### 3. Запустите проект

```bash
docker compose build
docker compose up -d
docker compose ps
```

Если все хорошо:

- образ успешно собирается;
- запускается один сервис `kerio-connect`;
- `docker compose ps` показывает контейнер как `Up`;
- health state может недолго быть `starting`, затем должен стать `healthy`.

Если подключаете боевой или внешний источник:

- по умолчанию не направляйте боевой почтовый трафик в эту лабораторию;
- для проверки логов вручную настройте Kerio external Syslog в UI администрирования Kerio после первичной настройки;
- сначала используйте тестовый или лабораторный Syslog-приемник, например связанный `kerio-logstash-project`.

### 4. Проверьте результат

Проверьте состояние контейнера:

```bash
docker compose ps
```

Если все хорошо:

- сервис `kerio-connect` находится в состоянии `Up`;
- health state равен `healthy`.

Проверьте открытые порты:

```bash
ss -ltn '( sport = :4040 or sport = :443 or sport = :25 or sport = :465 or sport = :587 or sport = :993 or sport = :995 )'
```

Если все хорошо:

- есть listeners для портов, опубликованных через `.env`;
- при настройках по умолчанию лаборатория слушает `4040`, `443`, `25`, `465`, `587`, `993` и `995`.

Проверьте сервис Kerio внутри контейнера:

```bash
docker compose exec -T kerio-connect /usr/local/bin/healthcheck.sh && echo HEALTHCHECK_OK
docker compose exec -T kerio-connect /etc/init.d/kerio-connect status
```

Если все хорошо:

- первая команда печатает `HEALTHCHECK_OK`;
- вторая команда печатает `Kerio Connect is running..`.

Проверьте начальный веб-доступ:

```bash
wget --server-response --spider http://localhost:4040/admin/ 2>&1 | sed -n '1,20p'
wget --no-check-certificate --server-response --spider https://localhost:4040/admin/ 2>&1 | sed -n '1,20p'
```

Если все хорошо:

- во время первичной настройки `http://localhost:4040/admin/` может ответить первым;
- после первичной настройки Kerio Connect перенаправляет административный доступ на `https://localhost:4040/admin/`;
- HTTPS-ответ содержит `HTTP/1.1 200 OK`;
- в заголовках ответа есть `Server: Kerio Connect`.

### 5. Зафиксируйте итог

После шагов выше:

- локальная лаборатория Kerio Connect запущена;
- административный UI доступен на порту `4040`;
- постоянные Docker volumes существуют для state, store, license и logs;
- можно пройти вендорский мастер первичной настройки в браузере;
- после завершения первичной настройки нормальный вход администратора ожидается по адресу `https://localhost:4040/admin/`.

## Устранение неполадок

### Проблема: порт 25 уже занят

**Симптомы**

- `docker compose up -d` завершается с ошибкой bind для порта `25`.
- SMTP не запускается на ожидаемом порту хоста.

**Что проверить**

- Локальный MTA, например `postfix`, может уже слушать порт `25` на хосте.
- `KERIO_SMTP_PORT` может всё еще быть равен `25`.

**Как исправить**

```bash
systemctl stop postfix || true
sed -i 's/^KERIO_SMTP_PORT=25/KERIO_SMTP_PORT=2525/' .env
docker compose up -d
```

### Проблема: сборка не находит или не скачивает Kerio installer

**Симптомы**

- `docker compose build` падает на этапе поиска инсталлятора.
- Сборка сообщает, что Debian-инсталлятор Kerio Connect недоступен.

**Что проверить**

- В `artifacts/` может не быть официального Kerio Connect `.deb`.
- Хост может не иметь доступа к `cdn.kerio.com`.
- `KERIO_DOWNLOAD_URL` может быть пустым или некорректным.

**Как исправить**

```bash
ls -la artifacts
grep -E '^(KERIO_AUTO_DOWNLOAD|KERIO_VERSION_LABEL|KERIO_DOWNLOAD_URL|KERIO_DOWNLOAD_SHA256)=' .env
docker compose build --no-cache
```

Затем используйте один из вариантов:

- поместите один официальный Kerio Connect `.deb` в `artifacts/`;
- укажите корректный официальный `KERIO_DOWNLOAD_URL`;
- оставьте `KERIO_AUTO_DOWNLOAD=1` и убедитесь, что хост видит `cdn.kerio.com`.

### Проблема: административная страница не открывается

**Симптомы**

- `http://localhost:4040/admin/` или `https://localhost:4040/admin/` недоступен.
- Контейнер остается в состоянии `health: starting` или `unhealthy`.

**Что проверить**

- Контейнер может еще стартовать.
- `KERIO_ADMIN_PORT` может отличаться от `4040`.
- Kerio Connect мог не запуститься корректно внутри контейнера.

**Как исправить**

```bash
docker compose ps
docker compose logs --tail=200 kerio-connect
docker compose exec -T kerio-connect /usr/local/bin/healthcheck.sh
```

Если вы изменили `KERIO_ADMIN_PORT` в `.env`, проверяйте этот порт вместо `4040`.

### Проблема: встроенная trial-ссылка возвращает 404

**Симптомы**

- Мастер первичной настройки отправляет на старый URL `kerio.com`, который больше не работает.

**Что проверить**

- Встроенный UI Kerio Connect может всё еще ссылаться на устаревший trial URL вендора.

**Как исправить**

Используйте текущую ручную точку входа для trial:

```text
https://gfi.ai/products-and-solutions/email-and-messaging-solutions/kerioconnect/free-trial
```

## Что проект не делает

- Он не является production-моделью развертывания Kerio Connect.
- Он не поставляет Kerio Connect installer или лицензионные материалы.
- Он не заменяет официальную документацию или поддержку GFI / Kerio.
- Он не автоматизирует весь вендорский мастер первичной настройки.
- Он не парсит, не нормализует, не обогащает и не анонимизирует Kerio logs; это роли других репозиториев семейства.
- Он не гарантирует долгосрочную доступность вендорского пакета при включенном auto-download; локальные артефакты или pinned URLs лучше для долгосрочной повторяемости.

## Что важно знать

- Kerio Connect — проприетарное стороннее ПО GFI / Kerio.
- Этот репозиторий оборачивает официальный вендорский пакет для лабораторного использования; он не меняет лицензию и не распространяет само ПО Kerio Connect.
- [LICENSE](./LICENSE) применяется к wrapper-коду, скриптам и документации этого репозитория, но не к вендорским бинарным файлам Kerio Connect.
- См. [THIRD_PARTY_NOTICE.md](./THIRD_PARTY_NOTICE.md) для явного разграничения стороннего проприетарного ПО.
- Во время первичной настройки admin UI может быть доступен через `http://localhost:4040/admin/`; после настройки ожидается редирект на `https://localhost:4040/admin/`.
- Для локальной первичной проверки публичная MX-запись не требуется.
- Рекомендуемый шаблон внешнего Syslog target для соседнего logging stack: `host.docker.internal:5514`.
- После успешной первичной настройки разумно сделать backup или snapshot ВМ перед дальнейшими экспериментами.
- Во время сборки лаборатория генерирует `en_US.UTF-8` и `ru_RU.UTF-8`, потому что Kerio administration ожидает наличие UTF-8 locales.

- английский язык остается основным языком проектной документации и ревью;
- обновления русского README приветствуются, если они сохраняют смысл английского README;
- русская документация помогает быстрее войти в проект и не меняет задокументированное поведение.

## GitHub Release Notes

GitHub Release Notes остаются на английском языке.

Пишите их для DevOps-инженеров, системных администраторов и лабораторных инженеров. Фокусируйтесь на том, что изменилось для человека, который запускает, проверяет или диагностирует проект:

- что пользователи теперь могут запускать, наблюдать, проверять или диагностировать;
- точные локальные проверки, run ID, статус CI и ожидаемые pass/fail-сигналы, когда они подтверждают готовность;
- нужные действия инженера, изменения конфигурации, миграционные заметки, известные ограничения, ручные шаги или superseded release notes.

Избегайте внутренних или пофайловых формулировок, если они не меняют способ использования проекта инженерами.
